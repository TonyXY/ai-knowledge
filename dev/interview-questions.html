AI 面试题 ​

面试 AI 相关岗位（AI 应用开发、AI 架构、Prompt 工程师），除了常规的编程题，还会考察对 AI 模型的理解深度和工程落地能力。

以下按主题分类整理。

一、LLM 基础 ​

Q1：解释一下 GPT 模型中的「自注意力机制」是什么？ ​

自注意力（Self-Attention）是 Transformer 的核心。它让模型在处理一个词时，能同时"关注"输入序列中的所有其他词，计算它们之间的相关性权重。

通俗类比：阅读时用荧光笔划重点——每个词都在问"这句话里哪些词和我最相关？"，然后根据相关度加权汇总。

关键词：QKV（查询/键/值）、注意力分数、Softmax 归一化、加权求和。

Q2：Transformer 为什么比 RNN 好？ ​

RNNTransformer
串行计算（慢）并行计算（快）
长距离依赖弱自注意力捕捉全局
梯度消失/爆炸残差连接 + LayerNorm
难以扩展可堆叠多层

Q3：什么是「上下文窗口」？为什么重要？ ​

上下文窗口是模型单次能处理的最大 Token 数。它决定了：

  • 能输入多长的文档
  • 多轮对话的"记忆"长度
  • 代码审查时能看多少文件

目前主流：128K（GPT-4o）→ 200K（Claude）→ 1M（Gemini）/ 2M（Kimi）。

Q4：什么是「幻觉」？如何缓解？ ​

幻觉指模型生成看似合理但事实错误的内容。

缓解方法：

  • RAG：检索外部知识库作为上下文
  • Prompt 约束：要求模型标注不确定的内容
  • 温度调低：减少随机性
  • 多模型验证：用另一个模型交叉检查
  • Fine-tuning：在特定领域数据上微调

二、Prompt 工程 ​

Q5：System Prompt 和 User Prompt 的区别？ ​

System PromptUser Prompt
作用设定模型行为、角色、规则具体任务指令
优先级通常高于 User Prompt每次对话不同
用户可见通常隐藏用户输入的内容

好的 System Prompt 包含：角色定义 + 行为规则 + 输出格式 + 边界条件。

Q6：什么是 Few-Shot / Zero-Shot？什么时候用 Few-Shot？ ​

  • Zero-Shot：不给示例，直接让模型做（"把这个句子翻译成英文"）
  • Few-Shot：给 2-3 个示例，再让模型做

Few-Shot 适合：

  • 输出格式复杂（如 JSON 结构特定）
  • 任务风格不明确（如特定写作风格）
  • 模型初次接触的任务类型

Q7：什么是思维链（Chain-of-Thought）？为什么有效？ ​

CoT 是引导模型逐步推理再给出答案的技巧。它有效的原因：

  • 把复杂问题分解为中间步骤
  • 每一步的结果作为下一步的上下文
  • 减少"直接跳结论"导致的错误

经典用法：在 Prompt 末尾加 Let's think step by step.

Q8：什么是结构化输出（JSON Mode）？怎么实现？ ​

让模型输出可解析的 JSON，而不是自由文本。

python
response = client.chat.completions.create(
    model="gpt-4o",
    response_format={"type": "json_object"},
    messages=[{"role": "user", "content": "提取姓名和电话：张三，13800138000"}]
)

三、RAG ​

Q9：解释一下 RAG 的完整流程？ ​

  1. 索引阶段：文档 → 分块 → Embedding → 存入向量数据库
  2. 检索阶段：用户问题 → Embedding → 向量检索 Top-K
  3. 生成阶段：问题 + 检索结果 → LLM → 回答

Q10：RAG 的常见优化手段有哪些？ ​

  • 分块策略：递归分块 + overlap
  • Re-ranking：向量检索 Top-50 → 精排取 Top-5
  • Hybrid Search：向量相似度 + 关键词 BM25 融合
  • Query Rewriting：把模糊问题改写为更精确的查询
  • Multi-hop RAG：先检索再追问再检索

Q11：什么是 Re-ranking？为什么必要？ ​

Re-ranking 是第二阶段精排。向量检索只做"语义相似"匹配，但不一定最相关。Re-ranker 用交叉编码器对问题和候选文档逐一打分，精度更高。

收益：提升 15-30% 检索准确率。

四、Agent ​

Q12：什么是 AI Agent？和普通 LLM 调用的区别？ ​

普通 LLM 调用：用户问 → 模型答（一次性）

Agent：模型能自主决策、调用工具、观察结果、循环执行直到完成任务。

text
Agent 循环（ReAct）：
Thought → Action → Observation → Thought → ... → Answer

Q13：解释 Tool Calling / Function Calling 的流程？ ​

  1. 开发者定义 Tool（名称、描述、参数 Schema）
  2. 用户提问，模型判断需要调用哪个 Tool
  3. 模型返回 Tool 名称和参数（JSON）
  4. 开发者执行函数，把结果返回给模型
  5. 模型基于结果生成最终回答

Q14：Multi-Agent 是什么？和单 Agent 的区别？ ​

单 Agent：一个模型完成所有任务 Multi-Agent：多个 Agent 分工协作，每个 Agent 专注一个领域

优势：

  • 专业化（每个 Agent 有自己的 System Prompt 和 Tool）
  • 可扩展（加新能力 = 加新 Agent）
  • 容错（一个 Agent 失败不影响整体）

五、工程实践 ​

Q15：如何评估 LLM 的输出质量？ ​

方法说明
人工评估最准确，但慢且贵
LLM-as-Judge用 GPT-4 评估 GPT-3.5 的输出
自动化测试用预期输出做断言匹配
BenchmarkMMLU、HumanEval 等标准化评测

Q16：如何控制 LLM API 的成本？ ​

  • 模型选择：简单任务用便宜的模型（GPT-4o-mini vs GPT-4o）
  • 缓存：相同或相似请求命中缓存
  • Batch API：批量请求有折扣
  • Token 预算：设置 max_tokens 上限
  • Prompt 压缩：精简上下文，去除冗余

Q17：如何防御 Prompt 注入攻击？ ​

  • 定界符：用明确标记包裹用户输入
  • 输入过滤：拦截已知攻击模式
  • 输出校验：检查输出是否包含敏感信息
  • 权限最小化：模型 Tool 只给最低必要权限
  • 模型选型：Claude 在安全方面表现更好

Q18：本地模型部署用什么方案？各自优缺点？ ​

六、开放问题 ​

Q19：你如何为某个任务选择模型？ ​

考虑维度：

  • 任务复杂度：简单问答 → 轻量模型，复杂推理 → 旗舰模型
  • 延迟要求：实时交互 → 低延迟模型，离线批处理 → 高精度模型
  • 预算：高频调用 → 高性价比模型
  • 数据隐私：敏感数据 → 本地模型或私有 API
  • 语言：中文为主 → 国产模型

Q20：如果 LLM 回答质量不够好，你会怎么排查和优化？ ​

排查链路：

  1. Prompt 是否清晰？→ 优化 Prompt
  2. 上下文是否充足？→ 增加相关上下文
  3. 模型是否选对？→ 换更强的模型
  4. 是否需要额外知识？→ 接入 RAG
  5. 是否需要特定能力？→ 使用 Tool Calling
  6. 输出格式是否约束？→ 使用 JSON Mode
